우리가 지금까지 만들어 온 모든 시각화 자료에는 한 가지 공통점이
있었습니다. 분명 깔끔하고 훌륭하지만, 모두 정적인 이미지였다는 점입니다.
잘 찍은 한 장의 사진처럼 특정 순간의 정보를 완벽하게 보여주지만, 보는
사람이 더 깊이 탐색하거나 다른 각도에서 볼 수는 없었죠.

만약 보고를 받는 사람이 이런 질문을 던진다면 어떨까요?

"이 그래프에서 가장 눈에 띄는 저 점은 어떤 제품인지 알 수 있을까?"

"전체적인 흐름은 알겠는데, 각 데이터의 세부정보도 함께 볼 수 있을까?"

그러면 엑셀 원본을 다시 열거나 추가자료를 찾아 헤매야 합니다.

이때 우리는 '상호작용하는[Interactive]{.m1} 시각화'를 통해 이 문제를
해결할 수 있습니다. 지금까지의 시각화가 한 편의 '잘 찍은 사진'이었다면,
인터랙티브 시각화는 '사용자가 직접 탐험할 수 있는 다큐멘터리 영상'과
같습니다. 사진은 감독이 의도한 장면만 보여주지만, 다큐멘터리 영상은
시청자가 원하는 부분에서 잠시 멈춰 확대해 보거나 관련 정보를 클릭해 더
깊이 알아볼 수 있는 것과 같죠.

이 마법은 파워포인트나 미리캔버스 같은 일반적인 디자인 툴로는 구현이
거의 불가능하며, AI 코딩의 가치를 가장 극적으로 보여주는 영역입니다.

\

#### [정교한 시각화를 지시하는]{.u2} {#032.xhtml_sigil_toc_id_35 title="정교한 시각화를 지시하는 ‘VIEW’ 프레임워크"}

['VIEW' 프레임워크]{.u2}

\

이런 정교한 결과물을 얻으려면 우리의 '지시'도 더 정교해져야 합니다.
AI에게 정교한 시각화를 지시하기 위한 VIEW 프레임워크를 소개합니다.

\

::: tb1
[VIEW 프레임워크]{.cor1}

\

**V[(Visualization Goal, 시각화 목표)]{.m}** : 무엇을 '강조'하고
싶은가?[(예 : 매출성장률을 강조하고 싶어)]{.m}

**I[(Input Data, 입력 데이터)]{.m}** : 어떤 '재료'를 사용할 것인가?[(예
: 지난 12개월 월별 매출 데이터 사용)]{.m}

**E[(Interaction Element, 상호작용 요소)]{.m}** : 사용자가 '무엇을' 하면
'어떤 일'이 일어나게 할 것인가?[(예 : 마우스를 올리면 세부정보가
나타나게 해줘)]{.m}

**W[(Workflow & Style, 흐름과 스타일)]{.m}** : 전체적인 시각적
'흐름'이나 '규칙'은 무엇인가?[(예 : 매출이 높을수록 원이 커지게
해줘)]{.m}
:::

\

이 프레임워크를 사용하면, AI는 우리의 머릿속에 있는 복잡한 그림을 훨씬
더 정확하게 구현해 낼 수 있습니다.

그럼 이제 AI와 대화하며 **\[서울 인구변화 시계열 애니메이션 버블
차트\]**를 만들어 보겠습니다. 보고받는 사람이 버튼 하나만 누르면 연도별
인구가 자연스럽게 변하며 버블의 크기ㆍ위치가 움직이는 형태를 만드는
겁니다. 정적인 차트로는 절대 구현할 수 없는, AI 코딩이 만들어 내는 진짜
마법을 체험해 보겠습니다.

\

당신은 도시계획 전문가입니다. 지난 수십 년간 서울 시내 주요 구역의
인구변화를 시간 흐름에 따라 드라마틱하게 보여주는 발표자료를 만들어야
합니다. 단순히 숫자만 나열하는 것이 아니라, 마치 영상처럼 '재생' 버튼을
누르면 연도별 인구가 역동적으로 변화하는 모습을 보여주고
싶습니다[(제미나이 Pro 모드 사용을 권장합니다)]{.m}.

\

**1단계. 무엇을 만들고 싶은지 상상하기(V : 시각화 목표)**

우리의 목표는 '특정 지역의 인구가 시간에 따라 어떻게 변화하고
이동했는지'를 시각적인 스토리텔링으로 보여주는 겁니다. 특히 '재생'
버튼을 누르면 움직이는 애니메이션 효과를 통해 연도별 데이터가
'과거-현재-미래'의 흐름으로 한눈에 파악되게 하는 겁니다.

\

**2단계. 제미나이에게 VIEW 프레임워크로 지시하기(I, E, W)**

이제 우리의 모든 요구사항을 담아, 제미나이 캔버스 모드로 지시를
내립니다. 제미나이 캔버스의 실시간 시각화 기능을 활용해 AI와의
대화만으로 이 모든 것을 구현해 보겠습니다.

\

#### [인터랙티브 차트 활용하기]{.u2} {#032.xhtml_sigil_toc_id_36}

\

AI와의 대화를 통해 완성된 이 '살아있는 차트'는, 우리의 보고서를 완전히
다른 차원으로 바꿔 줍니다. 정적인 차트로는 전달할 수 없었던 '변화의
흐름'과 '추세'를 효과적으로 보여줄 수 있습니다.

\

**(1) 발표자료의 '하이라이트' 영상으로 활용**

이 HTML 파일을 웹 브라우저에서 열고, 애니메이션 재생 모습을 화면
녹화하여 발표자료에 삽입해 보세요. 복잡한 설명을 줄이고, 강력한 시각
효과로 청중의 몰입도를 최고조로 끌어올릴 수 있습니다.

\

**(2) 웹 페이지, 데이터 대시보드에 직접 삽입**

생성된 HTML 코드를 사내 인트라넷, 웹 보고서, 혹은 데이터 대시보드에 바로
삽입할 수 있습니다. 사용자들이 직접 '재생'하며 데이터를 탐색하고
인사이트를 얻도록 유도할 수 있습니다.

\

**(3) 심층분석의 출발점으로 활용**

특정 시점의 급격한 인구변화를 발견했다면, 해당 시점을 멈춰 놓고 추가적인
분석을 시작하는 출발점으로 삼을 수 있습니다.

단순한 '영상'이 아니라 그 자체로 완벽한 '분석 도구'가 되도록 '연결'

Part 3의 여정을 마친 여러분께 한 가지 질문을 던지고 싶습니다. '코딩을
잘한다'는 것은 과연 어떤 의미일까요? 많은 사람들은 '파이썬 문법을
암기하고, 검은 화면에 막힘없이 코드를 써 내려가는 능력'이라고
생각합니다. 하지만 바로 이런 오해가 코딩을 향한 가장 큰 편견이자, 넘기
힘든 벽처럼 느껴지게 만드는 이유입니다.

'하수'는 코딩을 컴퓨터에게 던지는 '일방적인 지시'라고 착각합니다. 그래서
AI가 짜준 코드를 실행했을 때, 빨간색 에러 메시지가 뜨면 이렇게 생각하죠.

'AI가 엉터리 코드를 줬네, 역시 못쓰겠어.'

'뭐가 문제인지 하나도 모르겠어. 역시 코딩은 나랑 안 맞아.'

그들은 에러[Error]{.m1}를 '실패' 또는 '장벽'으로 인식하고, 거기서 대화를
멈춰 버립니다. 마치 자판기에 돈을 넣었는데 음료수가 나오지 않으면
자판기를 두어 번 걷어차고는 포기해 버리는 것과 같습니다.

반면, '고수'는 코딩을 AI와 함께 답을 찾아가는 '대화'라고 생각합니다.
그들에게 에러 메시지는 '실패'가 아니라 AI가 보내는 또 다른 '질문'일
뿐입니다. 고수는 이 에러 메시지를 그대로 복사해 붙어넣고 다시 AI에게
질문합니다.

'방금 네가 짜준 코드를 실행하니 이런 에러가 발생했어. 이게 무슨 뜻이고,
어떻게 해결하면 될까?'

이것이 바로 고수와 하수를 가르는 결정적 차이입니다. 하수는 에러 앞에서
포기하고, 고수는 에러를 가지고 다시 대화합니다.

고수는 AI를 완벽한 정답만 내놓는 '마법 상자'로 보지 않습니다. 오히려
'컴퓨터 언어에 매우 능통하지만, 업무의 구체적인 맥락은 아직 배우는 중인
아주 똑똑한 신입사원'으로 대하죠. 우리는 신입사원이 첫 업무에서 실수를
했다고 바로 해고하지 않습니다. 무엇을 잘못 이해했는지 다시 설명해 주고,
해결책을 함께 찾아 나갑니다.
